Legacy workflow

AI coding workflow for maintaining old projects

Old projects are rarely difficult for one reason. The challenge is usually a mix of stale dependencies, unclear conventions, missing tests, forgotten deployment steps, and code that still works but no longer explains why it works.

An AI coding assistant can help with maintenance, but it should not be pointed at the whole repository and told to modernize everything. The safer workflow is slower at the start: map the project, choose a narrow change, add a verification path, then edit in small steps.

Map the project before editing

The first AI task should be read-only. Ask the assistant to identify the main entry points, the folders that matter, the test commands, and the areas that look generated or risky. The output should be a short map, not a refactor plan.

For an older repository, this map is useful even when it is incomplete. It tells you where the assistant thinks the boundaries are, which gives you a chance to correct bad assumptions before code changes begin.

Separate maintenance from modernization

Maintenance and modernization are different jobs. Maintenance keeps the project working. Modernization changes the underlying shape of the project. Mixing both in one change makes review harder and increases the chance of breaking behavior users already depend on.

Ask for a behavior summary

Before changing old code, ask the assistant to summarize what the current code does in plain language. Then compare that summary against what you know about the product. This catches a common failure mode: plausible edits based on a wrong model of the system.

For example, a billing helper may look redundant until you notice it supports an old customer plan. A route may look unused until it is called by a scheduled job. A strange condition may be guarding data from a previous migration.

Add a verification path before the patch

If the project has weak tests, do not start by asking for a broad test suite. Start with one verification path for the behavior being touched:

  • A small unit test for a pure function.
  • A regression test for a bug that has already happened.
  • A local browser check for a visible workflow.
  • A command that proves the app still builds after the change.

This gives the assistant a concrete target and gives you a way to reject changes that look reasonable but do not prove anything.

Keep prompts narrow and reversible

Old projects benefit from narrow prompts. Ask for one file, one behavior, or one dependency at a time. If the assistant needs to touch more files, it should explain why before making the wider change.

A useful instruction is: "Make the smallest reviewable change and leave unrelated cleanup for a separate pass." That keeps maintenance from turning into a hidden rewrite.

Review for compatibility, not just correctness

New code can be correct and still incompatible with the old project. During review, check for changed public APIs, modified data shapes, renamed environment variables, altered URLs, and different default behavior.

This is where an AI review pass can help. Ask it to focus on regressions and compatibility risks rather than style. If it cannot point to a specific behavior that changed, treat the finding as low confidence.

Document what you learned

Every maintenance task should leave the repository a little easier to work with. Add a note to the README, a command to a script section, or a short comment near a non-obvious compatibility branch. Do not document everything. Document the thing that would have saved time at the start of the task.

This is how old projects become easier to maintain without a large rewrite. Each small fix also improves the context available to the next AI-assisted change.
