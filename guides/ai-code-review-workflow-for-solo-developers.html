Review workflow

AI code review workflow for solo developers

Solo developers do not get the natural second pair of eyes that a team review provides. That is where an AI coding assistant can help, but only if the review is structured. Asking "is this code good?" usually produces shallow feedback. A better workflow gives the assistant a narrow job, useful context, and a clear standard for what counts as a real issue.

Start with a small review target

AI review works best when the change is small enough to reason about. Before asking for feedback, split unrelated edits into separate commits or pull requests. A styling cleanup, a data model change, and a payment flow fix should not be reviewed as one pile.

For a solo project, a practical review target is one feature, one bug fix, or one refactor with a short explanation of intent. If the diff is too large to explain in three sentences, it is probably too large for one AI review pass.

Give the assistant the job a reviewer would have

A strong prompt is specific about risk. Instead of asking for general suggestions, assign a review role:

  • Bug review: Look for logic errors, edge cases, null states, async failures, and broken assumptions.
  • Security review: Check user input, authorization, secrets, unsafe redirects, and data exposure.
  • Regression review: Compare the changed behavior against the old behavior and call out compatibility risks.
  • Test review: Identify the smallest tests that would prove the change works.

Do these as separate passes. A single broad review tends to blur important issues into generic comments.

Use a repeatable review prompt

Keep a small prompt in the repository so review quality does not depend on memory. This version is intentionally plain:

Review this change as a senior engineer.
Focus on bugs, regressions, missing tests, and unclear behavior.
Ignore style preferences unless they affect maintainability.
List findings first, with severity and file references.
If you find no real issues, say that and mention remaining test gaps.

The instruction to list findings first matters. It pushes the assistant away from long summaries and toward actionable review output.

Ask for tests after issues, not before

Many developers ask an AI assistant to write tests immediately. That can be useful, but the better sequence is to review the risk first and then ask for tests that target the highest-risk behavior.

For example, if the review finds that a date parser may fail around timezone boundaries, the test request should focus on timezone cases. If the risk is an empty state, the test should cover missing data. The goal is not to increase test count; the goal is to cover the behavior most likely to break.

Keep the human decision in the loop

AI review is not an approval system. Treat it like a fast reviewer that can be wrong. Read every finding, reproduce suspicious claims, and reject comments that do not match the code.

A useful rule is simple: do not change code only because the assistant suggested it. Change code because the issue is real, the behavior is unclear, or the test gap is worth closing.

Run a final shipping check

Before merging or deploying, run a short checklist:

  1. Does the diff only contain the intended change?
  2. Did the AI review produce any unresolved bug or regression findings?
  3. Were the focused tests, lint checks, or build checks run locally?
  4. Is there a rollback path if the change affects production data or payments?

This is where AI assistance becomes valuable for a solo developer: not by replacing judgment, but by making review discipline easier to repeat.
