Repository setup

How to prepare a repository for an AI coding assistant

An AI coding assistant is easier to use when the repository explains itself. Clear scripts, small docs, predictable folders, and focused tests give the assistant better context and make its changes easier for the developer to review.

The goal is not to write a large internal manual. The goal is to remove ambiguity from the parts of the project that change often: setup, architecture, commands, conventions, and safety boundaries.

Start with the commands a reviewer needs

Every repository should make the common commands obvious. A coding assistant can usually inspect package files, but it should not have to infer which command is safe or current.

Command typeWhat to documentWhy it helps
InstallThe exact package manager and install commandPrevents accidental lockfile or dependency churn
Run locallyThe dev server command and expected localhost URLMakes browser verification faster
TestUnit, integration, and focused test commandsLets the assistant verify narrow changes
Format or lintThe command used before committingReduces noisy style-only diffs
BuildThe production build or static export commandCatches packaging and deployment issues early

Put these in the README or a short contributor note. If a command is slow or flaky, label it instead of letting the assistant discover that by trial and error.

Add a small project map

A project map gives the assistant enough orientation to avoid editing the wrong layer. It can be as short as five bullets:

  • Where UI components live.
  • Where data access or API clients live.
  • Where shared types, schemas, or validation rules live.
  • Where tests and fixtures live.
  • Which folders are generated, vendored, or not meant for manual edits.

This is especially useful in older projects where folder names no longer match the current architecture. The map does not need to be perfect. It only needs to reduce the chance of confident edits in the wrong place.

Write down the boundaries

AI assistants are better when they know what not to touch. Add a short section for boundaries such as:

  • Do not change database migrations without explicit review.
  • Do not edit generated files directly.
  • Do not change public API response shapes unless the task asks for it.
  • Do not rewrite unrelated modules while fixing a narrow bug.
  • Do not remove existing user changes to make a patch cleaner.

These rules are not only for the assistant. They also help the solo developer keep review focused when work moves quickly.

Keep examples close to the code

If the project has patterns that matter, place one clear example near the implementation. Examples are often more useful than long style documents. A test fixture, a small route handler, or a known-good component gives the assistant a local pattern to follow.

This matters most for forms, error handling, API clients, authentication checks, and background jobs. Those areas usually have conventions that are easy to break with a plausible-looking edit.

Make tests easy to run in slices

A solo developer does not always have time to run a full suite after every small change. Give the assistant focused paths:

  • A command for one test file.
  • A command for one package or workspace.
  • A command for changed frontend behavior, if the project has browser tests.
  • A clear note for tests that require secrets, services, or seeded data.

Focused tests make AI edits more reliable because the assistant can verify the behavior it touched instead of guessing from static inspection.

Add a review prompt to the repository

If AI review is part of the workflow, keep the review standard in the repo. A short file like docs/ai-review.md can tell the assistant to prioritize bugs, regressions, missing tests, and risky behavior over subjective style comments.

This keeps the workflow consistent across tools. Whether the developer uses a code editor assistant, a chat assistant, or a command-line agent, the review criteria stay the same.

Prepare before the repository gets large

The best time to prepare a repository is before it becomes hard to explain. Small notes added early prevent repeated context setup later. For a solo developer, that means less time restating the same project rules and more time reviewing actual changes.

A prepared repository does not make AI work automatic. It makes AI work reviewable. That is the practical difference between quick suggestions and changes that can safely move toward production.
